# Overview Dashboard 本地联调验收清单（Task9）

> 适用范围：仅用于 `overview dashboard` 本地联调验收。  
> 验收对象：`/ui` 页面与其依赖的数据链路（WS/SSE/接口聚合）。  
> 执行结论：每一步必须记录 `PASS/FAIL`，任一关键项失败则整体验收失败。

**通过判定**
- 页面可访问且无 5xx 阻断错误。

**失败判定**
- 页面无法打开、持续白屏、关键资源请求失败导致功能不可用。

## 1. `/ui` 首屏渲染与可视完整性检查

### 执行步骤

1. 首次进入 `/ui`，观察 10 秒内是否完成主要区域渲染（顶部、主图、侧边统计/列表等）。
2. 检查布局是否错位：标题、卡片、图例、筛选控件是否在可视区域内。
3. 检查字体、颜色、间距是否明显异常（如重叠、截断、闪烁）。
4. 检查 Console 是否有阻断性报错（`Uncaught`, `TypeError`, 资源加载失败导致组件不渲染）。

**通过判定**
- 首屏在可接受时间内完成渲染（建议 <= 3 秒，视本机性能可浮动）。
- 主要 UI 结构完整，无明显错位/重叠/大片空白。
- 无阻断主流程的前端异常。

**失败判定**
- 首屏超时不出图、结构缺失、主区域空白。
- 有阻断性错误导致关键模块不可见或不可用。

---

## 2. 拓扑缩放 / 拖拽 / 筛选 / 链路高亮检查

### 执行步骤

1. **缩放**：滚轮放大/缩小拓扑图，观察节点与连线是否同步缩放、无抖动错位。
2. **拖拽**：按住画布拖拽视图，检查是否平滑且无明显卡死；释放后位置保持。
3. **筛选**：依次切换可用筛选项（状态/类型/关键字等），确认节点数量与列表联动变化。
4. **链路高亮**：点击节点或链路触发高亮，检查高亮路径是否连续、非目标链路是否正确弱化。
5. **恢复默认**：清空筛选并重置视图，确认可回到初始展示状态。

**通过判定**
- 交互动作连续可用，不出现错误状态残留。
- 筛选结果与可视图一致，链路高亮逻辑正确。

**失败判定**
- 缩放/拖拽失效或严重卡顿不可操作。
- 筛选不生效、数据不更新、链路高亮错乱或不可恢复。

---

## 3. WS 断线重连与 SSE 兜底检查

### 执行步骤

1. 正常加载 `/ui` 后，在 Network 中定位 WS 连接（如 `/ws` 或 `/api/.../ws`）。
2. 人工断网 5-10 秒后恢复网络，观察：
   - WS 是否自动重连；
   - 页面数据是否恢复增量更新。
3. 人工阻断 WS（例如后端暂时禁用 WS 路由或通过代理拦截 WS），保留 HTTP 通道。
4. 观察是否自动切换到 SSE（EventSource）兜底拉流；检查数据是否继续更新。
5. 恢复 WS 后观察是否回切/保持稳定（以当前实现为准），且无重复数据风暴。

**通过判定**
- WS 中断可在可接受时间内自动恢复，且页面无需手动刷新。
- WS 不可用时，SSE 兜底生效并可持续更新关键数据。
- 重连/切换过程中无明显重复渲染、无连续报错刷屏。

**失败判定**
- WS 中断后不重连、需手动刷新才能恢复。
- WS 不可用且 SSE 未接管，数据停止更新。
- 出现大量重复消息、数据抖动或错误刷屏。

## 4. 图标加载完整性（无 broken image）

### 执行步骤

1. 首屏完成后，目视检查所有图标与图片区域是否存在“裂图/占位图/空白方块”。
2. 在 Console 执行以下脚本检查图片加载状态：

```javascript
(() => {
  const broken = [...document.images].filter(img => !img.complete || img.naturalWidth === 0);
  console.table(broken.map(img => ({ src: img.src, alt: img.alt })));
  return { total: document.images.length, broken: broken.length };
})();
```

3. 在 Network 过滤 `Img`，确认无关键图标资源 404/500。

**通过判定**
- `broken` 数量为 `0`。
- 无关键图标资源请求失败。

**失败判定**
- 任一关键图标 broken。
- 图标请求存在稳定 4xx/5xx 且影响界面识别。

## 5. 数据正确性抽样对比（alerts / monitoring / resources / logs）

### 抽样原则

- 至少抽样 3 组关键数字（建议：告警数、资源总量、活跃监控项、最近日志计数）。
- 对比来源：`/ui` 概览卡片 vs 对应业务页面（`alerts`、`monitoring`、`resources`、`logs`）。

### 执行步骤

1. 记录 `/ui` 首页关键数字（截图或表格）。
2. 依次打开对应页面，记录同时间窗口的关键数字。
3. 对比差异：
   - 若有缓存/聚合延迟，记录允许误差与时间窗口（例如 1-2 分钟）。
4. 刷新 `/ui` 与业务页面各一次，确认数字趋势一致。

### 记录模板

| 指标 | `/ui` 数值 | 对应页面数值 | 允许误差 | 结果 |
|---|---:|---:|---|---|
| alerts_total |  |  | 0 或说明 | PASS/FAIL |
| monitoring_active |  |  | 0 或说明 | PASS/FAIL |
| resources_total |  |  | 0 或说明 | PASS/FAIL |
| logs_recent |  |  | 0 或说明 | PASS/FAIL |

**通过判定**
- 抽样指标与对应页面一致，或在声明的延迟窗口内可收敛一致。

**失败判定**
- 指标长期不一致且无合理延迟解释。
- 刷新后仍出现方向性错误（例如一增一减、数量级明显错误）。
