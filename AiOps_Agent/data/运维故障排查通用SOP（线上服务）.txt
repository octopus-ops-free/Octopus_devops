运维故障排查通用SOP（线上服务）

0. 先做边界确认
- 影响范围：单实例、单服务、单机房还是全站
- 影响对象：读请求、写请求、支付链路、核心交易链路
- 故障时间：首次出现时间，是否与变更时间重叠

1. 告警优先
- 先看告警面板，确定触发指标（CPU、内存、磁盘、错误率、时延）
- 核对告警级别和持续时间，避免仅依据瞬时尖峰误判

2. 服务状态
- 检查服务是否 running / degraded / crashloop
- 对比发布前后实例状态、重启次数、健康探针失败次数

3. 日志定位
- 先检索 ERROR/WARN，再按 traceId 或 requestId 关联上下游
- 重点关注：超时、连接池耗尽、依赖服务不可达、配置加载失败

4. 资源瓶颈
- CPU 持续高于 85% + run queue 增长 -> 可能为计算瓶颈
- 内存持续高位 + OOM 日志 -> 可能存在泄漏或配置过低
- 磁盘使用率高于 80% -> 日志或临时文件清理策略异常

5. 网络连通
- ping 通不代表应用可用，仍需 tcp 端口联通校验
- 跨可用区链路重点看延迟抖动和丢包

6. 常见处置顺序
- 优先止损：限流、降级、熔断、扩容
- 再修复：回滚配置/版本、重建连接池、修正错误参数
- 最后复盘：补告警、补仪表盘、补演练脚本

7. 风险控制
- 高危操作（重启/回滚/扩缩容）必须二次确认
- 生产操作必须记录操作人、变更单号、执行时间
- 禁止在工单与日志中明文暴露密码、密钥、Token
